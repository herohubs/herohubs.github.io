La maggior parte degli sviluppatori dedica molto tempo a costruire cosa chiedere a un modello AI. Pochi si fermano a pensare a che forma debba avere la risposta. È un errore. Il formato dell'output non è un dettaglio estetico. È parte della specifica.

Pensatelo come un contratto API. Quando progettate un endpoint decidete se restituisce JSON, XML o un blob binario. La scelta dipende da cosa dovrà fare il consumer con quei dati. La stessa logica vale per gli output AI. **Markdown e HTML sono due schemi diversi**, ciascuno ottimizzato per una pipeline di consumo differente.

##### Markdown: la rappresentazione intermedia

Markdown è un linguaggio di markup leggero, progettato per essere leggibile come testo semplice e compilabile in HTML. Dal punto di vista ingegneristico è una **rappresentazione intermedia**, un formato pensato per essere trasformato o modificato a valle. Quando un modello restituisce Markdown, vi consegna un payload facile da parsare, confrontare, versionare e passare ad altri strumenti: un word processor, un CMS, un template email, un repository `git`.

Questa è la sua forza. È anche il suo limite.

Markdown è intrinsecamente **lineare**. Si legge dall'alto verso il basso. Un titolo, un paragrafo, una lista, un'altra sezione. Quella struttura funziona perfettamente per documenti destinati alla lettura e alla modifica. Si inceppa nel momento in cui dovete esporre relazioni, confronti o strutture dati parallele, cose che richiedono un layout spaziale piuttosto che una narrativa sequenziale.

##### HTML: il target di rendering finale

HTML non deve essere compilato. È il target di rendering stesso. Quando un modello produce HTML raw, non sta generando una bozza: sta generando una **superficie**. Qualcosa che si apre nel browser, si condivide con un link, si inserisce in una dashboard, o si consegna direttamente a uno stakeholder che non leggerà mai un blocco di codice.

Con HTML il modello guadagna accesso alla dimensione del layout. Può affiancare alternative, costruire un indice navigabile, trasformare una lista piatta di eventi in una timeline, aggiungere sezioni espandibili per i dettagli, evidenziare i punti critici senza sperare che il lettore li trovi da solo in un muro di testo.

Non si tratta di decorazione. **Il layout è una forma di information architecture.**

##### Lo shift verso gli artefatti nei sistemi agentici

Questa distinzione diventa critica man mano che i modelli smettono di produrre risposte e iniziano a produrre **artefatti**. Una risposta risponde a una domanda e vive dentro la chat. Un artefatto è un oggetto autonomo che vive fuori dalla chat: un piano operativo, una matrice dei rischi, una dashboard comparativa, un piccolo strumento per elaborare dati.

Quando l'output è un artefatto, la linearità di Markdown diventa un vincolo strutturale. Se uno stakeholder deve esaminare quattro opzioni strategiche in parallelo, o identificare scadenze critiche senza leggere ogni paragrafo, il formato sequenziale lavora attivamente contro la comprensione.

HTML rimuove quel vincolo. Permette al modello di codificare le informazioni nello stesso modo in cui lo farebbe un'interfaccia ben progettata, con gerarchia, affordances e navigabilità integrate fin dall'inizio.

##### Un framework decisionale pratico

La scelta deriva direttamente da come verrà consumato l'output:

| Pattern di consumo | Formato consigliato |
| :--- | :--- |
| Leggere, modificare, copiare in un altro strumento | Markdown |
| Condividere, presentare o consultare senza modifiche | HTML |
| Alimentare una pipeline o una API a valle | Markdown o JSON |
| Mostrare a stakeholder non tecnici | HTML |

Il punto di decisione è sempre la stessa domanda: **il consumer modificherà questo output, o lo esplorerà?**

##### Delegare la decisione al modello

Quando la risposta non è chiara, potete delegare la scelta del formato al modello stesso:

```
Prima di rispondere, valuta quale formato è più adatto al mio obiettivo.
Se il risultato deve essere letto, modificato o copiato in un altro strumento, usa Markdown.
Se deve essere consultato, presentato o usato come documento autonomo, usa HTML statico.
Spiega la tua scelta in una frase, poi produce l'output.
```

Questo prompt fa due cose contemporaneamente. Vi obbliga a riflettere sul contesto di consumo prima di eseguire la query. E produce un modello che ragiona sulla **forma** oltre che sul contenuto, il che tende a migliorare la qualità complessiva dell'output.

##### Il formato dell'output fa parte della specifica

Gli ingegneri sanno bene che gli stessi dati possono essere serializzati in modi diversi, e che la scelta della serializzazione è rilevante. Un prompt non è diverso. "Cosa voglio sapere?" è solo metà della specifica.

L'altra metà è: che forma deve avere la risposta, chi la consumerà, e come?

Sbagliare quella seconda metà non produce un output errato. Produce un output che non verrà mai usato.
