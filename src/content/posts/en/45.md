Most developers spend a lot of time crafting what they ask an AI model. Far fewer think about what shape they want the answer in. That is a mistake. The output format is not a cosmetic detail — it is part of the specification.

Think of it the way you think about an API contract. When you design an endpoint, you decide whether it returns JSON, XML, or a binary blob. You choose based on what the consumer needs to do with the data. The same logic applies to AI outputs. **Markdown and HTML are two different schemas**, each optimized for a different consumption pipeline.

##### Markdown: the intermediate representation

Markdown is a lightweight markup language designed to be readable as plain text and compilable to HTML. From an engineering standpoint, it is an **intermediate representation** — a format meant to be transformed or edited downstream. When a model returns Markdown, it is handing you a payload that is easy to parse, diff, version-control, and pipe into other tools: a word processor, a CMS, an email template, a `git` repository.

That is its strength. It is also its limit.

Markdown is inherently **linear**. You read it top to bottom. A heading, a paragraph, a list, another section. That structure works perfectly for documents meant to be read and modified. It breaks down the moment you need to expose relationships, comparisons, or parallel data structures — things that require spatial layout, not sequential narrative.

##### HTML: the final rendering target

HTML does not need to be compiled. It is the rendering target itself. When a model produces raw HTML, it is not generating a draft — it is generating a **surface**. Something you open in a browser, share with a link, drop into a dashboard, or hand directly to a stakeholder who will never read a fenced code block.

With HTML, the model gains access to the full layout dimension. It can place alternatives side by side, build a navigable index, turn a flat event list into a timeline, add collapsible sections for details, and highlight critical items without relying on the reader to spot them buried in a wall of text.

This is not decoration. **Layout is a form of information architecture.**

##### The artifact shift in agentic systems

This distinction becomes critical as models move from producing responses to producing **artifacts**. A response answers a question and lives inside the chat. An artifact is a standalone object that lives outside it — an operational plan, a risk matrix, a comparison dashboard, a mini-tool for processing data.

When the output is an artifact, Markdown's linearity becomes a structural constraint. If a stakeholder needs to review four strategic options simultaneously, or scan for critical deadlines without reading every paragraph, sequential formatting actively works against comprehension.

HTML removes that constraint. It lets the model encode information the same way a well-designed interface would — with hierarchy, affordances, and navigability built in from the start.

##### A practical decision framework

The choice follows directly from how the output will be consumed:

| Consumption pattern | Recommended format |
| :--- | :--- |
| Read, edit, copy, paste into another tool | Markdown |
| Share, present, or browse without modification | HTML |
| Feed into a downstream pipeline or API | Markdown or JSON |
| Display to non-technical stakeholders | HTML |

The decision point is always the same question: **will the consumer edit this, or experience it?**

##### Delegating the decision to the model

When the answer is genuinely unclear, you can delegate the format choice itself:

```
Before generating your response, evaluate which output format best fits my goal.
If the result needs to be read, edited, or copied into another tool, use Markdown.
If it needs to be browsed, presented, or used as a standalone document, use static HTML.
Explain your choice in one sentence, then produce the output.
```

This prompt does two things at once. It forces you to think about the consumption context before executing the query. And it produces a model that reasons about **form** alongside content — which tends to improve overall output quality.

##### The output format is part of the spec

Engineers are comfortable with the idea that the same data can be serialized in multiple ways, and that the serialization choice matters. A prompt is no different. "What do I want to know?" is only half the specification.

The other half is: what shape should the answer take, who will consume it, and how?

Getting that second half wrong does not produce a wrong output. It produces an output that will never be used.
